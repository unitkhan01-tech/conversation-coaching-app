# V09 로컬 프로토타입 리뷰

## 1. 현재 점수

현재 로컬 프로토타입은 약 40점 / 100점으로 평가한다.

v0.1~v0.8보다 발전했지만, 아직 실제 제품 수준은 아니다.

## 2. 좋아진 점

현재 버전에서 좋아진 점은 아래와 같다.

- 답장 후보가 단순 말투 중심에서 관계 목표 중심으로 이동했다.
- 흐름 유지 / 주도권 회복 / 선 긋기라는 목적 차이가 생겼다.
- 상대 메시지의 기능을 일부 구분하기 시작했다.
- “톡을 왜 이렇게 많이 해!”처럼 핀잔성 메시지를 별도 상황으로 처리하기 시작했다.
- 이전보다 성숙한 상담문 느낌은 줄었다.
- “다른 답장 보기”로 반복성을 일부 줄였다.

## 3. 여전히 부족한 점

현재 버전의 한계는 아래와 같다.

- 문장 품질이 여전히 불안정하다.
- 실제 사람이 보낼 만한 문장과 AI 예시문 사이를 오간다.
- 하드코딩 후보 풀이라 입력이 조금 달라지면 맥락을 놓칠 수 있다.
- 관계 목표를 완전히 수행하지는 못한다.
- 상대의 뉘앙스, 말투, 공격성, 애매함을 깊게 읽지 못한다.
- 답장 후보가 상황에 따라 살아 움직이는 느낌은 부족하다.
- 사용자가 원하는 “관계의 흐름을 내가 원하는 방향으로 가져가는 대화 설계”에는 아직 부족하다.

## 4. 현재 버전의 정확한 지위

현재 버전은 아래처럼 정의한다.

- 로컬 프로토타입
- 관계 목표형 답장 구조 실험
- 하드코딩 기반 품질 한계 확인용
- AI 전환 전 중간 단계
- 제품 MVP 아님
- 출시 가능 버전 아님

## 5. 하드코딩 방식의 한계

현재 app.js는 후보 문장 풀에서 문장을 선택하는 방식이다.

이 방식은 아래 한계를 가진다.

- 입력 문장이 다양해질수록 대응력이 떨어진다.
- 상대 메시지의 미묘한 뉘앙스를 읽기 어렵다.
- 사용자의 관계 목표를 깊게 반영하기 어렵다.
- 문장 후보가 반복된다.
- 새로운 상황을 매번 사람이 추가해야 한다.
- 인간적인 즉흥성과 맥락성이 부족하다.

## 6. 다음 방향

다음 단계는 하드코딩 후보를 계속 늘리는 것이 아니다.

다음 단계는 AI 기반 관계 목표형 답장 생성 구조로 전환하는 것이다.

단, API 키를 브라우저 코드에 넣으면 안 된다.  
따라서 서버 또는 서버리스 구조가 필요하다.

## 7. 결론

현재 버전은 40점짜리 로컬 프로토타입으로 인정한다.

이제 목표는 40점을 45점으로 만드는 것이 아니라, AI 기반 구조를 통해 70점 이상을 노리는 것이다.
